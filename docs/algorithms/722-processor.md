# ABT-7.2.2-Processor: 찾기 쉬운 도움 정보 알고리즘 v1.1

### 🔗 KWCAG 2.2 Reference
- **7.2.2 (찾기 쉬운 도움 정보)**: 도움 정보가 제공되는 경우, 각 페이지에서 동일한 상대적인 순서로 접근할 수 있어야 한다.

---

이 알고리즘은 **KWCAG 7.2.2(찾기 쉬운 도움 정보)** 지침을 준수하며, 웹 사이트 내에서 사용자가 문제 해결을 위해 제공받는 연락처, FAQ, 도움말, 채팅 봇 등의 요소가 여러 페이지에 걸쳐 **시각적인 위치뿐만 아니라 마크업 순서 모두 일관성 있게 구현**되었는지 진단한다.

## 1. 대상 식별 및 데이터 수집 (Data Collection)

### [대상 요소 추출]
- 페이지 내 반복적으로 제공되는 다음 유형의 사용자 도움 정보 식별:
  1. **담당자 상세 연락처**: 전화번호, 이메일, 주소, 운영시간 등
  2. **담당자 연락 방법**: 메신저, 채팅창, 게시판, SNS 등
  3. **도움말 옵션**: FAQ, 사용법 등
  4. **자동화된 연결 방법**: 챗봇 링크 또는 버튼 등

### [데이터 속성 파악]
- 추출된 도움 정보 요소의 DOM 트리 내 마크업 순서(XPath 또는 상대적 순서 형상화).
- 추출된 도움 정보 요소의 시각적 렌더링 좌표(Bounding Box)와 화면 내 위치(예: 하단 footer, 우측 하단 고정).
- 현재 페이지 URL과 요소의 메타데이터.

## 2. 분석 및 분류 로직 (Analysis Pipeline)

### [단계 A] 다중 페이지 간 시각적 위치 일관성 분석
- 여러 페이지에 공통으로 나타나는 챗봇 팝업이나 Footer 연락처 정보의 **시각적 렌더링 좌표(위치)**가 일관되게 나타나는가?
- 결과: 페이지마다 도움 정보의 렌더링 위치가 명확하게 다를 경우 **[부적절]**.

### [단계 B] 다중 페이지 간 마크업 순서 일관성 분석
- 여러 페이지에 공통으로 나타나는 도움 정보의 **마크업 내 상대적 순서(DOM 트리 상의 위치)**가 일관되게 유지되는가?
- 결과: 시각적 위치는 비슷하지만, DOM 구조(마크업 순서) 상에서 각 페이지마다 위치가 다르면 스크린 리더 사용자에게 혼동을 주므로 **[부적절]**.

### [단계 C] 사이트 수준 종합 판단 (정성 평가)
- 자동화 도구가 완벽히 식별해내지 못하는 영역(예: A페이지는 FAQ, B페이지는 채팅으로 제공 방식이 달라진 경우 등)에 대해 전문가가 크로스 체킹을 수행한다.
- 결과: 자동화 지표상 문제가 없더라도 전문가 확인을 위해 **[검토 필요]** 상태로 안내.

## 3. 최종 상태 정의 (Final Status)
1. **부적절 (Warning):**
   - 2개 이상의 페이지 스캔 결과에서 도움 정보(전화번호, 이메일, 챗봇 등)의 **시각적인 위치**가 다르게 표시된 경우.
   - 2개 이상의 페이지 스캔 결과에서 도움 정보의 **마크업 순서(DOM 내 위치)**가 상이하게 나타나는 경우.
2. **검토 필요 (Review):**
   - 도움 정보 요소가 일관성 있게 탐지되었으나, 예외적인 레이아웃 변형 등으로 인해 전문가의 육안 크로스 체킹이 필요함. (도구는 발견된 요소의 스냅샷/경로를 하이라이트하여 제공)
3. **적절 (Pass):**
   - 반복되는 도움 정보가 웹 사이트 내 동일 위치(시각적 위치 및 마크업 순서 모두)에 일관성 있게 구현됨이 최종 확인됨.
4. **N/A:**
   - 해당 웹 사이트(또는 페이지 묶음)에 어떠한 형태의 도움 정보(연락처, FAQ 등)도 제공되지 않음. (제공 자체를 강제하는 지침은 아님)
