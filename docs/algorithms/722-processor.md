# ABT-7.2.2-Processor: 찾기 쉬운 도움 정보 알고리즘 v1.0

### 🔗 KWCAG 2.2 Reference
- **7.2.2 (찾기 쉬운 도움 정보)**: 도움 정보가 제공되는 경우, 각 페이지에서 동일한 상대적인 순서로 접근할 수 있어야 한다.

---

이 알고리즘은 **KWCAG 7.2.2(찾기 쉬운 도움 정보)** 지침을 준수하며, 웹 사이트 내에서 사용자가 문제 해결을 위해 제공받는 연락처, FAQ, 도움말, 채팅 봇 등의 요소가 여러 페이지에 걸쳐 일관된 위치나 시각적/구조적 순서로 제공되고 있는지 진단한다. 

## 1. 대상 식별 및 데이터 수집 (Data Collection)

### [대상 요소 추출]
- 페이지 내 도움 정보 제공 링크 및 버튼 식별:
  - 텍스트/aria-label 휴리스틱: "도움말", "FAQ", "고객센터", "연락처", "문의", "채팅", "help", "contact" 등.
  - 마크업 구조: `<nav>` 내의 도움말 링크, 하단 `<footer>` 내의 고객센터 영역, 우측 하단 고정(`position: fixed`) 챗봇 버튼 등.

### [데이터 속성 파악]
- 추출된 도움 정보 요소의 DOM 트리 내 구조적 경로(XPath 또는 CSS Selector 패턴).
- 추출된 도움 정보 요소의 시각적 렌더링 좌표(Bounding Box).
- (다중 페이지 분석 시) 현재 페이지 URL과 요소의 위치 메타데이터.

## 2. 분석 및 분류 로직 (Analysis Pipeline)

### [단계 A] 도움 정보 요소의 존재 여부 및 탐지
- 현재 페이지 내에 도움 정보 역할을 하는 요소가 하나 이상 존재하는가?
- (단일 페이지 검사 시 한계): 이 지침은 여러 페이지에 걸친 "일관성"을 평가하므로, 단일 페이지 스캔만으로는 자동 판별이 불가능하다.
- 결과: 도움 정보가 탐지되면, 이를 전문가가 **[사이트 전체에서 일관된 위치에 있는지]** 검토하도록 플래그를 설정.

### [단계 B] 사이트 수준의 일관성 비교 (정성 평가 / 다중 페이지 검사 시)
- 여러 페이지 스캔 데이터를 종합할 경우, 탐지된 도움 정보 요소가 각 페이지의 DOM 트리 상에서 동일한 부모 노드(예: `<header>`, `<footer>`) 하위에 있거나 동일한 상대적 순서를 갖는가?
  - 예: 모든 페이지에서 `<footer>` 내비게이션의 세 번째 항목으로 "고객센터" 링크 제공.
- 결과: 페이지마다 도움 정보 링크의 위치(DOM 순서 또는 시각적 위치)가 명확히 달라질 경우 **[수정 권고]**. 자동화 도구는 이를 확실히 판단하기 어려우므로 항상 **[검토 필요]** 상태로 안내.

## 3. 최종 상태 정의 (Final Status)
1. **검토 필요 (Review):**
   - 도움 정보 요소가 발견되었으나, 사이트 내 타 페이지와의 일관성 여부는 자동 검사로 완벽히 확정할 수 없어 전문가의 크로스 체킹이 필요함. (도구는 발견된 요소의 스냅샷/경로를 하이라이트하여 제공)
2. **수정 권고 (Suggestion):**
   - 2개 이상의 페이지 스캔 결과에서 도움 정보 요소의 상대적 위치나 DOM 구조가 상이하게 나타나는 경우.
3. **적절 (Pass):**
   - 전문가가 육안 및 구조적으로 사이트 전반에 걸쳐 일관된 위치에 도움 정보가 제공됨을 확인함. (전문가 판정 입력 시)
4. **N/A:**
   - 해당 웹 사이트(또는 페이지 묶음)에 어떠한 형태의 도움 정보(연락처, FAQ 등)도 제공되지 않음. (제공 자체를 강제하는 지침은 아님)
